# ADR-0001: Adoção da Clean Architecture

## Status

Aceito ✅

## Contexto

O projeto **Play Learn Grow / Plalegro App** é multiplataforma (Android + Web) e precisa de:

- Separação clara de responsabilidades.
- Testabilidade independente de frameworks.
- Escalabilidade para novas features e módulos.
- Consistência entre mobile e web, compartilhando lógica de negócio.

A arquitetura tradicional em camadas (UI → Service → DB) não atende bem a esses requisitos, pois acopla lógica de negócio a frameworks e dificulta testes isolados.

## Decisão

Adotamos a **Clean Architecture** como padrão, com as seguintes camadas:

- **Core (Domínio):**

  - Entidades, Value Objects e Casos de Uso.
  - Puro Kotlin, sem dependência de Android ou frameworks.
  - Retorna sempre `Result<Success, Failure>` para evitar estados inválidos.

- **App (Android):**

  - UI com Jetpack Compose.
  - Navegação com Navigation Compose.
  - Injeção de dependência com Hilt.
  - Integração com `core` para lógica de negócio.

- **Web (Next.js):**

  - UI moderna com Next.js.
  - Deploy contínuo em Vercel.
  - Integração futura com APIs que podem consumir lógica do `core`.

- **Infra (CI/CD):**
  - Pipelines GitHub Actions para CI, Staging e Produção.
  - Deploy automático para Vercel (Web) e Firebase (Android).

## Consequências

- **Positivas:**

  - Código mais limpo, modular e testável.
  - Facilidade de manutenção e evolução.
  - Reuso da lógica de negócio em diferentes plataformas.
  - QA e DevOps têm pipelines claros e automatizados.

- **Negativas:**
  - Curva de aprendizado inicial maior para novos desenvolvedores.
  - Necessidade de disciplina para manter separação entre camadas.
  - Mais arquivos e estrutura, o que pode parecer complexo no início.

## Alternativas consideradas

- **MVC tradicional:** rejeitado por acoplamento excessivo.
- **MVVM sem domínio separado:** rejeitado por misturar lógica de negócio na camada de apresentação.
- **Arquitetura hexagonal:** próxima da Clean Architecture, mas optamos pela Clean por maior familiaridade da equipe.

## Decisão final

A **Clean Architecture** será o padrão oficial do projeto, documentada e aplicada em todos os módulos.

# ADR-0006: Uso do GitHub Actions como orquestrador de CI/CD

## Status

Aceito ✅

## Contexto

O projeto **Play Learn Grow / Plalegro App** precisa de pipelines robustos para:

- **CI (Continuous Integration):** build, testes unitários, lint e análise estática.
- **CD (Continuous Deployment):** deploy automático em staging e produção.
- Integração com múltiplos módulos (Android, Next.js, Shared KMP, Ktor Server).
- Gestão centralizada de workflows e secrets.

Alternativas como **GitLab CI**, **CircleCI** e **Jenkins** foram consideradas.  
A equipe já utiliza GitHub como repositório principal, o que torna GitHub Actions uma escolha natural.

## Decisão

Adotar **GitHub Actions** como orquestrador principal de CI/CD:

- Workflows YAML versionados no repositório (`.github/workflows/`).
- Integração nativa com GitHub (pull requests, branches, secrets).
- Execução paralela de jobs (CI, lint, deploy web, deploy Android).
- Artefatos de build e relatórios de testes armazenados automaticamente.
- Integração com Vercel (Web) e Firebase App Distribution (Android).

## Consequências

- **Positivas:**
  - Configuração simples e declarativa (YAML).
  - Integração nativa com GitHub (sem ferramentas externas).
  - Suporte a ambientes separados (staging e produção).
  - Escalabilidade com runners hospedados ou self-hosted.
- **Negativas:**
  - Dependência da infraestrutura GitHub.
  - Limites de minutos de execução em planos gratuitos.
  - Menor flexibilidade comparado a soluções customizadas como Jenkins.

## Alternativas consideradas

- **GitLab CI:** robusto, mas exigiria migração de repositório.
- **CircleCI:** bom para projetos cloud-native, mas menos integrado ao GitHub.
- **Jenkins:** poderoso e flexível, mas exige manutenção de servidores e maior esforço de DevOps.

## Decisão final

O **GitHub Actions** será usado como orquestrador oficial de CI/CD.  
Todos os pipelines (CI, staging, produção) serão definidos em `.github/workflows/`, garantindo automação, consistência e integração com o ecossistema GitHub.
